# Technical Debt

## Inter-Service Protocol Inconsistency

- [ ] **AC internal APIs use HTTP/JSON without shared contracts**: All other inter-service communication (GC↔MC, GC↔MH, MC↔MH) uses gRPC with proto-defined contracts in `proto/internal.proto`. AC is the exception — its internal APIs (`/api/v1/auth/internal/meeting-token`, `/api/v1/auth/internal/guest-token`, `/api/v1/auth/token`) are HTTP/JSON with request/response structs defined independently on each side. This caused a production-visible bug: GC's `MeetingTokenRequest` defined `home_org_id: Option<Uuid>` while AC's expects `home_org_id: Uuid` (required), resulting in 422 errors at runtime. No compile-time check catches these mismatches. Options: (a) migrate AC internal APIs to gRPC and add to `internal.proto`, or (b) create a shared `ac-api-types` crate that both AC and GC import. The `common::token_manager` (OAuth token fetch) is already shared but the meeting/guest token request types are not. Locations: `crates/gc-service/src/services/ac_client.rs` (GC's structs), `crates/ac-service/src/models/mod.rs` (AC's structs).

- [ ] **`RejectionReason` lacks an `INVALID_REQUEST` variant**: `proto/internal.proto:189-194` defines `RejectionReason` as `UNSPECIFIED|AT_CAPACITY|DRAINING|UNHEALTHY`. When MC's `AssignMeetingWithMh` rejects a malformed request (e.g., empty `mh_assignments` or empty `MhAssignment.grpc_endpoint`), the handler at `crates/mc-service/src/grpc/mc_service.rs:248-251` maps the `McError::InvalidArgument` to `RejectionReason::Unhealthy`, which is misleading — MC is healthy, GC sent bad data. GC consumers may interpret this as a transient MC issue and retry, which won't help. Add `REJECTION_REASON_INVALID_REQUEST` and route invalid-argument paths to it. Low frequency (GC contract violations should never happen), but the right semantic for observability and correct GC behavior under bug conditions.

- [ ] **Dead `accepted: bool` field in three registration response messages**: `RegisterMHResponse` (`proto/internal.proto:231-235`), `RegisterMCResponse` (`:90-95`), and `RegisterMeetingResponse` (`:133-135`) all carry a `bool accepted = 1` field. Producer sites hardcode it `true` in every path: `crates/gc-service/src/grpc/mh_service.rs:176` (MH registration), `crates/gc-service/src/grpc/mc_service.rs:257` (MC registration), and the MH-side `RegisterMeeting` handler. The only error path in each is a typed `Err(Status::internal | ...)`; no conditional sets `accepted=false`. Consumer sites (`crates/mh-service/src/grpc/gc_client.rs:152-179, :296`; parallel sites in MC) have real branching logic that handles the unreachable `accepted=false` case and log-then-error. Surfaced during ADR-0032 Step 2 review when security noticed `mh_gc_registration_total{status=success}` was emitted unconditionally on `Ok(response)` regardless of `accepted`; initial framing was a P1 metric-labeling bug, but investigation showed the latent-not-active shape: the metric would be wrong *if* rejections happened, but GC never rejects. Resolution: remove the field from all three messages and delete the unreachable consumer-side branching. Constraints: (a) proto3 wire compat — a two-step rollout (reserve field + keep always-true first, delete in next release) avoids new-server × old-client skew issues where old clients would read the missing field as proto3 default `false` and fail registration. (b) 3 messages × 3 producers × 3 consumers is cross-service scope — protocol specialist ownership, likely wants a small /debate first to ratify the two-step. (c) Downstream: delete `gc_integration.rs::test_gc_client_registration_rejected` (tests an impossible state). Not urgent — metric bug cannot fire in production absent a GC-side code change that doesn't exist. Schedule after ADR-0032 Steps 3-5 complete so protocol changes don't overlap with MC/AC/GC metric-test backfills.

## Service Credential Management

- [ ] **AC admin API for service credentials**: Add `POST /api/v1/admin/service-credentials` to AC for registering/updating service credentials (client_id, scopes, service_type). Idempotent upsert. Currently service credentials are seeded via raw SQL in `infra/kind/scripts/setup.sh:seed_test_data()`, which has caused bugs (missing `internal:meeting-token` scope for GC) because credential config is decoupled from the code that requires it.
- [ ] **Per-service credential registration Jobs**: Each service (GC, MC, MH) should own its credential registration via a K8s Job in its Kustomize base. The Job calls the AC admin API to register the service's client_id and required scopes. Deploy pipeline ensures AC is ready before downstream services deploy. This way, adding a new scope (e.g., `internal:meeting-token`) is part of the same PR that adds the endpoint requiring it.
- [ ] **Remove setup.sh seed_test_data**: Once the admin API and registration Jobs exist, remove the raw SQL credential seeding from `infra/kind/scripts/setup.sh` and `infra/docker/postgres/init.sql`.
