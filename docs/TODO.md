# Technical Debt

## Inter-Service Protocol Inconsistency

- [ ] **AC internal APIs use HTTP/JSON without shared contracts**: All other inter-service communication (GCŌåöMC, GCŌåöMH, MCŌåöMH) uses gRPC with proto-defined contracts in `proto/internal.proto`. AC is the exception ŌĆö its internal APIs (`/api/v1/auth/internal/meeting-token`, `/api/v1/auth/internal/guest-token`, `/api/v1/auth/token`) are HTTP/JSON with request/response structs defined independently on each side. This caused a production-visible bug: GC's `MeetingTokenRequest` defined `home_org_id: Option<Uuid>` while AC's expects `home_org_id: Uuid` (required), resulting in 422 errors at runtime. No compile-time check catches these mismatches. Options: (a) migrate AC internal APIs to gRPC and add to `internal.proto`, or (b) create a shared `ac-api-types` crate that both AC and GC import. The `common::token_manager` (OAuth token fetch) is already shared but the meeting/guest token request types are not. Locations: `crates/gc-service/src/services/ac_client.rs` (GC's structs), `crates/ac-service/src/models/mod.rs` (AC's structs).

- [ ] **`RejectionReason` lacks an `INVALID_REQUEST` variant**: `proto/internal.proto:189-194` defines `RejectionReason` as `UNSPECIFIED|AT_CAPACITY|DRAINING|UNHEALTHY`. When MC's `AssignMeetingWithMh` rejects a malformed request (e.g., empty `mh_assignments` or empty `MhAssignment.grpc_endpoint`), the handler at `crates/mc-service/src/grpc/mc_service.rs:248-251` maps the `McError::InvalidArgument` to `RejectionReason::Unhealthy`, which is misleading ŌĆö MC is healthy, GC sent bad data. GC consumers may interpret this as a transient MC issue and retry, which won't help. Add `REJECTION_REASON_INVALID_REQUEST` and route invalid-argument paths to it. Low frequency (GC contract violations should never happen), but the right semantic for observability and correct GC behavior under bug conditions.
